# ADR-004: Sticky Rows & Columns

**Статус:** Accepted
**Дата:** 2026-01-31
**Контекст проекта:** Библиотека виртуального скролла
**Связанные ADR:**

* ADR-001: Virtualization Architecture
* ADR-002: Public API
* ADR-003: Scroll Anchoring

---

## Context

Библиотека виртуального скролла должна поддерживать сценарии с фиксированными (sticky) строками и столбцами:

* header / footer в таблицах;
* фиксированные колонки (например, первая колонка с названием);
* комбинации sticky строк и столбцов (corner cases);
* совместную работу со скроллом по двум осям и виртуализацией.

Sticky-элементы должны:

* всегда оставаться видимыми;
* не ломать виртуализацию;
* корректно работать с dynamic size и scroll anchoring;
* масштабироваться от простого списка до сложного грида.

---

## Problem

Наивная реализация sticky через CSS (`position: sticky`) в условиях виртуализации приводит к проблемам:

* sticky-элементы могут быть удалены из DOM;
* расчёт диапазонов виртуализации усложняется;
* возникают рассинхронизации offset’ов;
* сложнее поддерживать dynamic размеры и prepend данных.

Также отсутствует чёткое разделение ответственности:

* что виртуализируется;
* что фиксируется;
* какие элементы участвуют в расчётах scroll offset.

---

## Decision

Принято архитектурное решение:
**Sticky строки и столбцы реализуются как отдельные render-слои и исключаются из виртуализации.**

---

### 1. Модель слоёв (Layered Rendering Model)

Для 2D-сцен (Grid / Table) используется следующая структура:

```
┌───────────────────────────────┐
│ Sticky Top Rows               │
├───────────────┬───────────────┤
│ Sticky Left   │ Virtual Body  │
│ Columns       │               │
├───────────────┴───────────────┤
│ Sticky Bottom Rows            │
└───────────────────────────────┘
```

Дополнительно:

* угловые зоны (corner cells) — отдельные слои;
* каждый слой имеет собственный контейнер и transform.

---

### 2. Исключение sticky из виртуализации

Sticky-элементы:

* **не входят** в `AxisModel`;
* **не участвуют** в расчётах `VirtualRange`;
* **не учитываются** в prefix sum и size index;
* имеют собственный жизненный цикл рендера.

Виртуальное тело компенсирует их размеры через offset:

* `scrollTopEffective = scrollTop - stickyTopHeight`
* `scrollLeftEffective = scrollLeft - stickyLeftWidth`

---

### 3. Конфигурация sticky (публичный контракт)

Sticky задаются декларативно:

```ts
type StickyConfig = {
  top?: number;
  bottom?: number;
  left?: number;
  right?: number;
};
```

Где значения — количество строк / столбцов, закреплённых в соответствующей зоне.

Это решение:

* не привязывает API к конкретной разметке;
* позволяет легко комбинировать зоны;
* напрямую отражает архитектуру слоёв.

---

### 4. Поведение при скролле

* Sticky Top / Bottom реагируют только на горизонтальный скролл.
* Sticky Left / Right реагируют только на вертикальный скролл.
* Virtual Body реагирует на обе оси.

Для каждого слоя применяется свой `transform`:

* `translateX(-scrollLeft)`
* `translateY(-scrollTop)`
* либо их комбинация.

---

### 5. Corner-зоны (пересечение sticky)

При наличии одновременно sticky строк и столбцов:

* создаются отдельные corner-слои:

  * top-left
  * top-right
  * bottom-left
  * bottom-right
* corner-слои:

  * не скроллятся;
  * рендерятся один раз;
  * имеют приоритет по z-index.

---

### 6. Dynamic size и sticky

Sticky элементы:

* могут иметь динамические размеры;
* измеряются отдельно от виртуального тела;
* их суммарные размеры (`stickyTopHeight`, `stickyLeftWidth`) участвуют в компенсации offsets.

Изменение размеров sticky:

* **не требует scroll anchoring**;
* приводит только к пересчёту effective viewport для virtual body.

---

### 7. Scroll anchoring и sticky

Взаимодействие с ADR-003:

* sticky элементы **не могут быть якорем**;
* anchoring всегда выполняется относительно virtual body;
* prepend данных в virtual body не влияет на sticky зоны.

Это обеспечивает:

* стабильность якоря;
* отсутствие “двойной компенсации”.

---

### 8. Поддержка List vs Grid

#### VirtualList

* поддерживаются `top` и `bottom`;
* `left/right` игнорируются.

#### VirtualGrid / VirtualTable

* поддерживаются все зоны;
* corner-слои активны при необходимости.

---

## Consequences

### Положительные

* Чёткая и предсказуемая модель рендера.
* Простые расчёты виртуализации.
* Корректная работа с dynamic size.
* Отсутствие конфликтов со scroll anchoring.
* Масштабируемость под сложные таблицы.

---

### Отрицательные / Trade-offs

* Увеличение количества DOM-слоёв.
* Более сложная разметка и стили.
* Sticky не участвуют в overscan (рендерятся всегда).

---

## Alternatives Considered

### 1. CSS `position: sticky` внутри виртуализации

❌ Нестабильно при unmount элементов.

### 2. Включение sticky в AxisModel

❌ Сильно усложняет расчёты и anchoring.

### 3. Отдельная виртуализация sticky зон

❌ Не даёт практической выгоды, увеличивает сложность.

---

## Non-Goals

* Поддержка sticky произвольной формы (free layout).
* Sticky внутри ячеек virtual body.
* Автоматическое объединение header/footer логики.

---

## Testing Strategy

* Unit:

  * корректность расчёта effective viewport;
  * компенсация offsets.
* Integration:

  * вертикальный + горизонтальный скролл;
  * dynamic size sticky;
  * комбинации top + left + corner.
* Visual regression:

  * z-index и наложения;
  * синхронизация при быстром скролле.
