# Virtualization Library — Project Overview

## Назначение проекта

Данный проект представляет собой **библиотеку виртуального скролла**, предназначенную для эффективного отображения больших наборов данных в UI (списки, таблицы, гриды) с гарантированной производительностью и предсказуемым поведением.

Библиотека решает инфраструктурную задачу и может использоваться как базовый слой в UI-китах и продуктовых приложениях, где требуется работа с десятками и сотнями тысяч элементов без деградации UX.

---

## Ключевые возможности

* Виртуализация **строк и столбцов** (1D и 2D).
* Поддержка **фиксированных и динамических размеров** элементов.
* **Sticky строки и столбцы** (header, footer, fixed columns).
* **Программная прокрутка** к строке / столбцу / ячейке.
* Стабильный скролл без скачков за счёт **scroll anchoring**.
* Разделение алгоритмов виртуализации и UI-рендера.
* Поддержка сложных сценариев: prepend данных, late-measure, resize.

---

## [Архитектурный подход](./architecture.md)

### 1. Унифицированная модель виртуализации

Библиотека использует единую координатную модель:

* ось **Y** — строки;
* ось **X** — столбцы.

Любой список рассматривается как частный случай грида с одной колонкой.
Это позволяет использовать одинаковые алгоритмы для List, Grid и Table.

---

### 2. Разделение стратегий размеров

Поддерживаются две стратегии:

* **Fixed size** — для элементов с известным размером (O(1), предпочтительный режим).
* **Dynamic size** — для элементов с переменной высотой/шириной.

Dynamic-режим основан на структуре префиксных сумм (**Size Index**) и обеспечивает:

* поиск элемента по scroll offset;
* вычисление offset по индексу;
* логарифмическую сложность операций.

---

### 3. Scroll Anchoring как обязательная часть системы

Для предотвращения scroll jump’ов библиотека использует **явный anchor-based механизм**:

* фиксируется логическая позиция пользователя;
* при изменении размеров или данных scroll корректируется;
* работает для dynamic size, prepend, scrollToIndex, late-measure.

Anchoring является внутренним механизмом и прозрачен для потребителя API.

---

### 4. Sticky строки и столбцы как отдельные слои

Sticky элементы:

* не участвуют в виртуализации;
* рендерятся в отдельных слоях;
* компенсируются при расчёте effective viewport.

Такой подход:

* упрощает расчёты;
* исключает конфликт с anchoring;
* масштабируется для сложных таблиц с corner-зонами.

---

### 5. [Многоуровневый публичный API](./public-api.md)

API библиотеки разделён по уровням сложности:

1. **High-level компоненты**

   * `VirtualList`
   * `VirtualGrid`
   * `VirtualTable` (façade)

2. **Headless hooks**

   * `useVirtualList`
   * `useVirtualGrid`

3. **Imperative API**

   * `scrollToIndex`
   * `scrollToRow / scrollToColumn / scrollToCell`
   * `measure`, `getVisibleRange`

Это позволяет использовать библиотеку:

* как plug-and-play решение;
* как низкоуровневый engine для кастомных UI.

## Принципы проектирования

* **Производительность по умолчанию** — ограниченное количество DOM-нод.
* **Явность вместо магии** — оценки размеров задаются явно.
* **Архитектурная масштабируемость** — 1D и 2D используют один core.
* **Стабильность API** — SemVer, контролируемый депрекейт.
* **Чёткие границы ответственности** — core ≠ React ≠ consumer.

---

## Что библиотека НЕ делает

* Не решает бизнес-логику таблиц (сортировка, фильтрация, агрегации).
* Не виртуализирует произвольный free-layout (masonry).
* Не гарантирует pixel-perfect поведение без разумных `estimatedItemSize`.
* Не экспортирует внутренние структуры (AxisModel, SizeIndex) как public API.

---

## Целевая область применения

Библиотека подходит для:

* больших списков и логов;
* таблиц с фиксированными колонками;
* чатов и лент с prepend данных;
* BI-интерфейсов и data-heavy UI;
* UI-китов зрелых продуктов.

Не рекомендуется для:

* малых списков (< 100 элементов);
* сценариев, где требуется полный DOM;
* нестабильных auto-layout без ограничений.
