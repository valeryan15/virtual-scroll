# ADR-001: Virtualization Architecture

**Статус:** Accepted
**Дата:** 2026-01-31
**Контекст проекта:** Библиотека виртуального скролла (List / Grid / Table)
**Решение принято:** Frontend Architecture

---

## Context

В проекте требуется библиотека виртуального скролла для отображения больших наборов данных (десятки и сотни тысяч элементов) с предсказуемой производительностью и стабильным API.

Библиотека должна покрывать следующие сценарии:

* виртуализация строк и столбцов;
* поддержка фиксированных (sticky) строк и столбцов;
* работа с фиксированными и динамическими размерами элементов;
* программная прокрутка к заданной строке / ячейке;
* использование как в списках, так и в таблицах / гриде;
* масштабируемость архитектуры без переписывания базовых алгоритмов.

Ограничения и вводные:

* DOM-рендер должен оставаться ограниченным по количеству элементов;
* поведение скролла должно быть стабильным (без jump’ов);
* алгоритмы виртуализации не должны быть жёстко связаны с React;
* библиотека должна быть пригодна для долгосрочной эволюции (new layouts, new render layers).

---

## Problem

Наивный рендер больших списков приводит к:

* деградации производительности (layout, paint, memory);
* нестабильному скроллу при динамических размерах;
* сложности поддержки фиксированных строк/столбцов;
* высокой связности алгоритмов виртуализации и UI-рендера.

Также отсутствует единый архитектурный подход, позволяющий:

* одинаково работать с 1D и 2D сценами;
* комбинировать fixed и dynamic размеры;
* расширять функциональность без ломки API.

---

## Decision

Принята **слоистая архитектура виртуализации**, основанная на следующих принципах:

### 1. Унифицированная координатная модель

* Вся виртуализация описывается через оси:

  * Y — строки
  * X — столбцы
* Список (List) — частный случай Grid с `columns = 1`.

---

### 2. Разделение стратегий размеров

Для каждой оси используется одна из стратегий:

#### Fixed-size strategy

* Используется при фиксированном размере строки/столбца.
* O(1) вычисления.
* Предпочтительный режим по умолчанию.

#### Dynamic-size strategy

* Используется при переменных размерах.
* Хранит измеренные размеры и fallback-оценки.
* Основана на структуре префиксных сумм (Fenwick Tree / Segment Tree).
* Обеспечивает:

  * поиск индекса по scroll offset;
  * вычисление offset по индексу.

---

### 3. Независимая модель оси (Axis Model)

Каждая ось (строки / столбцы) описывается контрактом:

```ts
interface AxisModel {
  count: number;
  totalSize: number;
  getRange(offset: number, viewportSize: number): VirtualRange;
  getOffsetByIndex(index: number): number;
}
```

Это позволяет:

* независимо комбинировать fixed/dynamic оси;
* использовать одинаковую логику для вертикального и горизонтального скролла.

---

### 4. Scroll anchoring как обязательный механизм

Для предотвращения скачков скролла применяется **anchor-based коррекция**:

* фиксируется якорный элемент (index + offset внутри);
* при пересчёте размеров корректируется scrollTop / scrollLeft;
* механизм обязателен для:

  * dynamic размеров;
  * prepend данных;
  * late-measure (изображения, раскрывающиеся блоки).

---

### 5. Фиксированные (sticky) строки и столбцы как отдельные слои

* Sticky элементы **не участвуют в виртуализации**.
* Рендерятся в отдельных слоях:

  * top / bottom (строки);
  * left / right (столбцы).
* Виртуальное тело компенсирует их размеры через смещения.

Это упрощает:

* расчёты диапазонов;
* поддержку сложных таблиц;
* расширение sticky-логики в будущем.

---

### 6. Core ≠ UI (React-layer)

Архитектура разделена на слои:

```
Core (алгоритмы)
 ├─ Axis models
 ├─ Range calculation
 ├─ Size index
 └─ Anchor manager

UI layer (React)
 ├─ Scroll container
 ├─ Render window
 └─ DOM measurement
```

Core не зависит от React и может быть переиспользован.

---

### 7. Программный контроль скролла как часть API

Поддерживается API:

```ts
scrollToIndex(index, options);
scrollToCell(rowIndex, columnIndex);
```

Реализация:

* fixed mode → прямое вычисление offset;
* dynamic mode → через prefix sum;
* обязательно учитывает anchoring и ещё не измеренные элементы.

---

## Consequences

### Положительные

* Предсказуемая производительность независимо от размера данных.
* Унифицированная архитектура для List / Grid / Table.
* Простое добавление новых сценариев (например, virtual masonry).
* Чёткие границы ответственности между слоями.
* Минимизация scroll jump’ов.

---

### Отрицательные / Trade-offs

* Dynamic-режим сложнее в реализации и требует структуры данных O(log n).
* Initial totalSize в dynamic-режиме является оценочным.
* Sticky элементы увеличивают количество render-слоёв.
* Scroll anchoring усложняет логику скролла и тестирование.

---

## Alternatives Considered

### 1. Единый алгоритм без разделения fixed/dynamic

❌ Приводит к усложнению и деградации fixed-кейсов.

### 2. Полная зависимость от DOM-измерений

❌ Высокая стоимость layout, нестабильность.

### 3. React-only решение

❌ Сильная связность, плохая расширяемость.

---

## Non-Goals

* Не решается задача виртуализации произвольной free-layout сцены.
* Не гарантируется полная SSR-виртуализация (fallback render).
* Не обеспечивается сохранение DOM-элементов вне viewport.
