# CONTRIBUTING.md

## Введение

Спасибо за интерес к развитию библиотеки виртуального скролла 🙌
Этот документ описывает **правила участия**, архитектурные ограничения и процесс внесения изменений.

Проект является **инфраструктурной UI-библиотекой**, поэтому приоритеты здесь отличаются от продуктовой разработки:

- предсказуемость поведения,
- стабильность API,
- архитектурная целостность,
- контроль производительности и техрисков.

---

## Архитектурный контекст (обязательно к прочтению)

Перед началом работы необходимо ознакомиться с архитектурными решениями проекта:

- **ADR-001** — Virtualization Architecture
- **ADR-002** — Public API
- **ADR-003** — Scroll Anchoring
- **ADR-004** — Sticky Rows & Columns
- **ADR-005** — Dynamic Size Index Implementation

👉 Любые изменения, противоречащие этим ADR, **запрещены** без их обновления.

---

## Основные принципы проекта

### 1. Core ≠ UI

- Алгоритмы виртуализации живут в `core`.
- React-слой — это адаптер (рендер, refs, observers).
- Core **не зависит** от DOM и React.

### 2. Одна модель для List и Grid

- List — частный случай Grid (`columns = 1`).
- Дублирование алгоритмов под “особый” List запрещено.

### 3. Fixed и Dynamic — осознанный выбор

- Fixed size — приоритетный режим.
- Dynamic size используется **только при необходимости**.
- `estimatedItemSize` обязателен для dynamic.

### 4. Никакой скрытой магии

- Никаких авто-угадываний размеров.
- Все нестабильные места должны быть явно описаны в коде и документации.

---

## Что можно и нельзя делать

### Разрешено

- Добавлять новые возможности через **расширение конфигураций**.
- Оптимизировать алгоритмы при наличии измерений и тестов.
- Добавлять headless hooks и façade-компоненты.

### Запрещено

- Ломать публичный API без ADR и migration path.
- Экспортировать внутренние структуры (`AxisModel`, `SizeIndex`).
- Добавлять `O(n)` операции в scroll / resize hot-path.
- Использовать `position: sticky` как основной механизм sticky.
- Отключать scroll anchoring в dynamic-режиме.

## Процесс внесения изменений

### 1. Обсуждение

Для нетривиальных изменений:

- заведи issue или draft PR;
- опиши проблему и предполагаемое решение;
- укажи, какие ADR затрагиваются.

### 2. ADR (если требуется)

ADR обязателен, если:

- меняется архитектурный инвариант;
- добавляется новый фундаментальный механизм;
- меняется поведение публичного API.

Формат ADR: `Context → Problem → Decision → Consequences`.

---

## Требования к Pull Request

Каждый PR должен:

- [ ] соответствовать **Code Review Checklist** проекта;
- [ ] не нарушать ADR;
- [ ] содержать тесты для core-логики;
- [ ] обновлять документацию при изменении поведения/API.

Рекомендуемый размер PR:

- **до ~500 строк** изменений (без автогенерации).

---

## Тестирование

### Обязательно

#### Unit tests (core)

- range calculation;
- dynamic size index;
- findIndexByOffset;
- scroll anchoring математика.

#### Integration tests (React)

- dynamic size + late-measure;
- prepend без scroll jump;
- scrollToIndex в dynamic;
- sticky + virtual body.

PR без тестов для core-логики **не принимается**.

---

## Производительность

Контрибьютор обязан:

- избегать layout thrashing;
- батчить измерения (`ResizeObserver`);
- использовать `requestAnimationFrame` для scroll-обновлений;
- контролировать количество DOM-нод.

Если PR заявлен как `perf`:

- необходимо описание **что именно ускорено** и **за счёт чего**.

---

## Публичный API и совместимость

- Проект следует **SemVer**.
- Breaking changes — только в `major`.
- Deprecation:
  - минимум 1 минорный релиз;
  - задокументированный migration path.

Любые изменения API:

- должны быть явно описаны в PR;
- сопровождаются обновлением docs.

---

## Стиль кода

- TypeScript с явными типами.
- Минимум side-effects.
- Комментарии — для архитектурных решений, а не очевидного кода.
- Никаких `any` без обоснования.

---

## Инструменты и автоматизация

### Codex / AI-агенты

- Проект использует `AGENTS.md`.
- Execution policy зафиксирована в `codex/rules/*.rules`.
- Агентам запрещено обходить архитектурные ограничения.

### Локальная разработка

- Не используйте destructive команды вне sandbox.
- Следуйте правилам, описанным в `codex/rules`.

---

## Что проект НЕ делает (Non-Goals)

- Masonry / free-layout виртуализация.
- Бизнес-логику таблиц (sorting, filtering).
- Автоматическое угадывание размеров.
- Полную SSR-виртуализацию.

PR, пытающийся добавить эти вещи, будет закрыт.

---

## Финальное правило

> **Если изменение упрощает код, но ослабляет архитектурные гарантии — оно не принимается.**

Цель проекта — не минимальный код, а **устойчивый, предсказуемый и масштабируемый фундамент** для сложных UI.
