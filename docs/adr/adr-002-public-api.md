# ADR-002: Public API

**Статус:** Accepted
**Дата:** 2026-01-31
**Контекст проекта:** Библиотека виртуального скролла
**Связанные ADR:**

* ADR-001: Virtualization Architecture

---

## Context

После принятия архитектуры виртуализации (ADR-001) требуется зафиксировать **публичный API библиотеки**, который:

* покрывает основные сценарии (List / Grid / Table);
* поддерживает фиксированные и динамические размеры;
* позволяет программно управлять скроллом;
* остаётся стабильным при дальнейшем развитии архитектуры;
* подходит как для простого использования, так и для сложных кастомных интеграций.

API библиотеки является долгоживущим контрактом и должен минимизировать техриски:

* изменения внутренних алгоритмов не должны приводить к breaking changes;
* сложность должна масштабироваться по мере необходимости, а не навязываться сразу.

---

## Problem

Без чётко зафиксированного публичного API возникает риск:

* утечки внутренних архитектурных деталей (axis, size index, anchor);
* перегруженных компонентов с избыточными пропсами;
* невозможности переиспользования core-алгоритмов;
* нестабильности API при добавлении новых кейсов (sticky, 2D, dynamic).

Также необходимо решить:

* как совместить простоту использования и архитектурную гибкость;
* где проходит граница между декларативным и императивным API;
* каким образом поддерживать сложные сценарии (таблицы, кастомный DOM).

---

## Decision

Принята **многоуровневая модель публичного API**, разделяющая библиотеку на несколько уровней потребления.

---

### 1. High-level компоненты (primary API)

Предоставляются готовые React-компоненты:

* `VirtualList`
* `VirtualGrid`
* `VirtualTable` (façade, опционально)

Характеристики:

* декларативный API;
* скрывает детали виртуализации;
* подходит для 80–90% кейсов;
* стабилен и минимален по поверхности.

**Обоснование:**
Позволяет быстро внедрять виртуализацию без погружения в алгоритмы.

---

### 2. Headless hooks (advanced API)

Предоставляются хуки:

* `useVirtualList`
* `useVirtualGrid`

Характеристики:

* не навязывают DOM-структуру;
* возвращают только рассчитанные диапазоны и offsets;
* позволяют интеграцию с кастомной версткой (Table, Canvas, SVG).

**Обоснование:**
Закрывает сценарии, где high-level компоненты недостаточно гибки.

---

### 3. Imperative API (контролируемый доступ)

Через `ref` предоставляются методы:

* `scrollToIndex`
* `scrollToRow / scrollToColumn / scrollToCell`
* `getVisibleRange`
* `measure`

Характеристики:

* используется точечно;
* не заменяет декларативную модель;
* работает одинаково для fixed и dynamic размеров.

**Обоснование:**
Без императивного API невозможно корректно поддержать:

* навигацию;
* deep-linking;
* внешнее управление состоянием скролла.

---

### 4. Явная конфигурация размеров

API разделяет стратегии размеров:

```ts
{ sizeMode: 'fixed', itemSize }
{ sizeMode: 'dynamic', estimatedItemSize }
```

**Решение:**

* `estimatedItemSize` обязателен для dynamic-режима;
* библиотека не пытается «угадывать» размеры.

**Обоснование:**
Исключает магию и снижает риск scroll jump’ов.

---

### 5. Sticky как отдельный контракт

Фиксированные строки и столбцы:

* не участвуют в виртуализации;
* описываются через `sticky`-конфигурацию;
* рендерятся отдельными слоями.

**Обоснование:**
API напрямую отражает архитектурное решение ADR-001 и предотвращает неявные зависимости.

---

### 6. ScrollTo как часть публичного контракта

Поддерживаются операции:

* прокрутка к строке;
* прокрутка к столбцу;
* прокрутка к ячейке.

**Решение:**

* одинаковый API для fixed/dynamic;
* поддержка align (`start | center | end | nearest`);
* корректная работа с ещё не измеренными элементами.

---

## Consequences

### Положительные

* Чёткая сегментация сложности API.
* Возможность развития core без breaking changes.
* Поддержка как простых, так и сложных кейсов.
* Снижение когнитивной нагрузки на потребителей.
* Прямое соответствие архитектуре виртуализации.

---

### Отрицательные / Trade-offs

* Увеличение количества экспортируемых сущностей.
* Headless hooks требуют более высокого уровня экспертизы.
* Imperative API усложняет тестирование.

---

## Alternatives Considered

### 1. Единый универсальный компонент

❌ Перегруженный API, плохая масштабируемость.

### 2. Только headless API

❌ Высокий порог входа для большинства команд.

### 3. Полностью императивный API

❌ Ломает декларативную модель React.

---

## Non-Goals

* Предоставление low-level алгоритмов (AxisModel, SizeIndex) как public API.
* Поддержка нестандартных scroll-контейнеров без overflow.
* Решение бизнес-логики таблиц (сортировка, фильтрация).

---

## Stability & Evolution Policy

* **SemVer**: breaking изменения — только в major.
* Deprecation:

  * минимум 1 минорный релиз с предупреждениями;
  * документированный migration path.
* Новые возможности добавляются через расширение конфигураций, а не изменение сигнатур.
